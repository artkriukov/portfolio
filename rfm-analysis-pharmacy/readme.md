# RFM-анализ клиентской базы аптечной сети

Проект для приоритизации SMS-кампаний: кого удерживать, кого реактивировать, куда не тратить бюджет. Основная логика — в **SQL**, notebook — для визуализации и выгрузки списков.

## Бизнес-проблема

Аптечная сеть перед сезонным спадом выручки планирует SMS-рассылки, но **не может отправлять одинаковые сообщения всей базе**. Персональные кампании дороги, а без приоритизации бюджет уходит на клиентов с низким вкладом в выручку. Нужно понять, кто приносит основные деньги и где риск потери выручки максимален.

## Решение

1. **Собрали поведенческий срез** по транзакциям бонусных карт за 12 месяцев (`sql/data_overview.sql`).
2. **RFM-сегментация** — квартили по давности, частоте и сумме покупок → 5 сегментов (`sql/rfm_segmentation.sql`).
3. **ABC-анализ** — ранжирование клиентов по накопительной выручке: A = 80%, B = 15%, C = 5% (`sql/abc_summary.sql`).
4. **Матрица RFM × ABC** — пересечение поведения и ценности (`sql/rfm_abc_matrix.sql`).
5. **Количественная оценка рисков** — выручка под угрозой при оттоке 30% ценных клиентов (`sql/risk_summary.sql`).
6. **Списки для маркетинга** — готовые CSV по правилам кампаний (`sql/action_segments.sql` → `data/exports/`).

## Ключевые выводы

| Наблюдение | Цифра | Действие |
| :--- | :--- | :--- |
| Концентрация выручки | 35% клиентов → 80% выручки | Фокус удержания на классе A |
| Ядро бизнеса | «Чемпионы» + «Лояльные» = 52% базы, 86,5% выручки | VIP-коммуникации, программы лояльности |
| Зона риска | 84 клиента A в «Под угрозой» → ~85 тыс. ₽ при оттоке 30% | Персональная реактивация |
| Win-back | 64 клиента A в «Нельзя терять» → ~52 тыс. ₽ при оттоке 30% | Индивидуальный контакт |
| Апселл | 25 «Чемпионов» класса B | Рост среднего чека |
| Экономия бюджета | 953 «Потерянных» класса C | Только дешёвые массовые SMS |

## Структура SQL

```
sql/
├── _base.sql              # CTE-пайплайн: params → RFM → ABC
├── data_overview.sql      # Объём данных и период анализа
├── rfm_segmentation.sql   # Сводка по RFM-сегментам
├── abc_summary.sql        # Сводка по ABC-классам
├── abc_revenue.sql        # Клиентский уровень с RFM + ABC
├── rfm_abc_matrix.sql     # Матрица сегмент × класс
├── risk_summary.sql       # Риски и возможности в рублях
└── action_segments.sql    # Списки для кампаний
```

Запросы подключаются через `_base.sql` + файл с финальным `SELECT` (см. `read_sql_from_file` в notebook).

## Ограничения

- Учитываются только покупки с картой `2000%` (без карты клиент не идентифицируется).
- Окно анализа — 12 месяцев от последней даты в данных.
- Пороги RFM основаны на квартилях текущей базы, при смене сезона их нужно пересчитывать.
- Прогноз оттока (ML) вынесен в отдельный проект `churn-prediction-pharmacy.ipynb`.
